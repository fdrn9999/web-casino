# PixiJS 전환 계획서 — 게임 렌더링 전면 이관 + 베팅 UX 극대화

> 작성일: 2026-07-21 · 선행 문서: `2026-07-21-improvement-plan.md`(버그/UX/기능 로드맵)
> 목표: 4개 게임(블랙잭·바카라·룰렛·슬롯)의 **게임 화면 렌더링을 PixiJS(WebGL) 캔버스로 전환**하고,
> 베팅 게이머의 사용자 경험(반응속도·연출·터치 정확성)을 네이티브 게임 수준으로 끌어올린다.

---

## 0. 전환하는 이유 (현 DOM 방식의 실측 한계)

이번 세션 실측에서 확인된 DOM/CSS 방식의 구조적 한계:

1. **레이아웃 시프트로 인한 오클릭** — 바카라에서 베팅하면 베팅 스팟 버튼 안에 칩 스택이 생기며
   아래 컨트롤 행이 ~33px 밀림. 실측 중 "되돌리기" 버튼 클릭이 반복적으로 빗나감.
   빠르게 연타하는 베팅 게이머에게 치명적 UX 결함. 캔버스는 좌표가 고정된 씬 그래프라 원천 차단.
2. **CSS transition 기반 연출의 타이밍 불일치** — 카드 딜/휠 스핀이 `setTimeout`+CSS 전환의 조합이라
   프레임 단위 제어가 불가(중간 취소·역재생·동기화가 어려움). Pixi ticker는 프레임 콜백 기반.
3. **브라우저 제스처와의 간섭** — 더블탭 확대·스크롤·텍스트 선택이 게임 입력과 충돌.
   이번 세션에서 CSS/이벤트 하드닝으로 완화했지만, 캔버스 단일 입력 표면이 근본 해법.
4. **연출 확장성 한계** — 파티클(칩 샤워), 모션 블러 스핀, 셰이더 광택 등은 DOM으로 불가/고비용.

**전환하지 않는 것(불변 원칙)**
- 서버 권위 원칙: 게임 로직·정산·검증은 지금처럼 100% 서버(`*Runner.js`). Pixi는 **표현 계층만** 담당.
- 소켓 프로토콜: `table:state` 스냅샷 + ack 이벤트 구조 유지. 렌더러 교체가 서버에 영향 없어야 한다.
- 접근성 병행 표기: 카드 텍스트("A♠ 10♠ = 21")·베팅 내역 등 스크린리더가 읽을 DOM 레이어는 캔버스 밖에 유지.
- `prefers-reduced-motion` 전면 대응 유지(Pixi 씬에서도 연출 스킵 분기).

## 1. 아키텍처

```
┌───────────────────────────── Vue 앱 셸 (유지) ─────────────────────────────┐
│ 라우터 · 로비 · 인증 · 채팅 · HUD(잔고/한도) · 관리자 · 마이페이지 · 토스트 │
│  ┌──────────────── 게임 뷰(교체) ────────────────┐                        │
│  │  <PixiStage>  ← 캔버스 1장, 게임별 Scene 마운트  │   DOM 오버레이:       │
│  │   BlackjackScene / BaccaratScene /             │   카드 텍스트·에러·   │
│  │   RouletteScene / SlotsScene                   │   접근성 라이브영역    │
│  └────────────────────────────────────────────────┘                        │
│        ↑ 상태 구독(기존 useGameSocket 그대로)                                │
└──────────────────────────────────────────────────────────────────────────┘
```

- **PixiJS v8** (WebGL2/WebGPU 자동, 폴백 canvas2d). `pixi.js` 단일 패키지.
- **PixiStage.vue**: 캔버스 생성·리사이즈(devicePixelRatio 대응)·씬 수명주기만 담당하는 공용 컴포넌트.
  - `touch-action: none` + Pixi 포인터 이벤트(`pointerdown/move/up`) → 마우스·터치 입력 통합
    (이번 세션 모바일 하드닝의 10번 항목이 자연 해결됨).
- **Scene 계층**(게임당 1개): 서버 스냅샷 → 씬 상태 diff → 트윈 큐 재생.
  기존 BlackjackView의 "progressive reveal 큐"(순차 공개 설계)를 Scene으로 그대로 이식 — 검증된 설계 재사용.
- **에셋 파이프라인**: 카드 SVG 52장+뒷면 → 스프라이트시트(texture atlas, 2x 해상도), 칩 6액면,
  휠/펠트/슈 텍스처. `assets/atlas/` 생성 스크립트(빌드 시 1회).
- **사운드**: 기존 `useSound.js`(WebAudio 합성) 그대로 사용 — 교체 불필요.
- **트윈**: 자체 마이크로 트위닝(ticker 기반 lerp + easing 함수 몇 개)으로 시작, 필요 시 gsap 검토.

## 2. 단계별 전환 로드맵

### Phase A — 공통 기반 (선행, 게임 미변경)
| 작업 | 내용 | 완료 기준 |
|---|---|---|
| A-1 | `pixi.js` 도입 + `PixiStage.vue`(마운트/리사이즈/파괴, reduced-motion 플래그 주입) | 데모 씬이 4:3~21:9 어디서든 왜곡 없이 리사이즈 |
| A-2 | 카드/칩 스프라이트시트 생성 파이프라인 | 아틀라스 1~2장, 메모리 < 32MB |
| A-3 | 공용 게임 오브젝트: `CardSprite`(플립 애니메이션), `ChipStackSprite`(액면 병합·두께), `ChipTraySprite`, `TimerBar` | 스토리 데모에서 딜/플립/칩 적층 재생 |
| A-4 | 입력 계층: 포인터 통합·히트영역(모바일 최소 44px)·롱프레스/드래그 제스처 유틸 | 터치·마우스 동작 동일(실기기 실측) |

### Phase B — 블랙잭 (가장 연출이 많음 → 기준 구현)
- 씬: 펠트·아치·슈, 7좌석, 딜러 영역(카드 나란히 배치·홀카드 플립·셔플 루프·슈 왕복 — 이번 세션에
  DOM으로 구현한 연출을 Pixi 트윈으로 승격), 좌석 캐스케이드, 승리 칩 샤워 파티클.
- 베팅 UI: 칩 트레이 → 베팅 스팟 **드래그 앤 드롭**(탭도 유지), 스테이징 되돌리기/지우기/최대/직전 베팅.
- 완료 기준: 기존 기능 100% 패리티(테스트 시나리오 표로 검증) + 60fps(미드레인지 모바일) + 접근성 DOM 병행.

### Phase C — 바카라
- 씬: 플레이어/뱅커 카드 순차 공개(스퀴즈 연출 옵션 — 카드 모서리 드래그로 천천히 까기),
  베팅 스팟 5종, 칩 애니메이션(베팅→스팟 비행, 환불→트레이 복귀).
- **중국점 로드는 DOM 유지**(이번 세션 구현한 `BaccaratRoads.vue` — 표 형태 정보라 DOM이 더 적합).
- 완료 기준: 패리티 + 스퀴즈 연출 온/오프 설정.

### Phase D — 룰렛
- 씬: 휠 물리(감속 곡선·공 낙하 바운스·포켓 안착을 ticker로 프레임 제어 — 현 CSS 전환의 타이밍 어긋남 해소),
  베팅 보드(인사이드 핫스팟을 히트영역 폴리곤으로 — DOM 오버레이 핫스팟보다 정확), 칩 스택 렌더.
- 완료 기준: 패리티 + 결과 번호와 공 안착 포켓 100% 일치(현 구현의 각도 계산 로직 이식·검증).

### Phase E — 슬롯 + 마무리
- 씬: 릴 스핀 블러·심볼 바운스·잭팟 연출. 로비/관리자/마이페이지는 DOM 유지(전환 대상 아님).
- 구 DOM 게임 뷰 제거, 번들 사이즈 점검(pixi ~450KB gzip 예산 반영, 게임 라우트 lazy-load 유지).

### 리스크와 대응
- **번들 증가** → 게임 라우트만 dynamic import(현 구조 유지), 아틀라스는 HTTP 캐시.
- **저사양 WebGL 실패** → Pixi 자동 canvas 폴백 + "간소 모드"(연출 최소화) 설정.
- **접근성 저하** → 캔버스는 장식, 정보는 DOM 병행 레이어가 항상 원본(카드 텍스트·베팅 내역·결과).
- **회귀 위험** → 게임별 패리티 체크리스트(베팅/정산/재접속/관전/채팅/한도)를 Phase 완료 게이트로.

## 3. 베팅 게이머 UX 극대화 백로그 (Pixi 전환과 병행)

### 이번 세션(2026-07-21)에 이미 완료 ✅
- 베팅 **되돌리기(undo)·전체 취소** — 서버 `bet:undo`/`bet:clear` + 즉시 환불, 룰렛·바카라 적용(E2E 실측),
  블랙잭 스테이징 되돌리기(클라). 기존 계획서의 U-05·F-08 해소.
- **블랙잭 최대(맥스벳) 버튼 + 잔고 클램프**, 딜러 카드 나란히 배치, 홀카드 플립, 슈 셔플 연출(사운드 포함),
  셔플↔슈 왕복 트랜지션.
- **바카라 중국점**(빅로드·빅아이·소로·커크로치, 표준 규칙) + 서버 히스토리 80판 보관.
- **모바일 터치 하드닝**: 게임용 viewport(핀치줌/더블탭 확대 차단), `touch-action` 정책(게임 표면 none,
  룰렛 보드 pan-x), 텍스트 선택·롱프레스 콜아웃·탭 하이라이트·드래그 제거, 멀티터치 차단(passive:false),
  overscroll 차단. 입력 필드는 선택 허용 예외.
- 유니코드 카드 글리프 → 폰트 안전 표기, 칩 시인성(두께 렌더·크기·라벨) 상향.

## 4. 실행 순서 제안

1. **X-1, X-2, X-6** — Pixi 전환 전에도 효과 큰 저비용 픽스(레이아웃 시프트·마감 레이스·테이블 안내).
2. **Phase A** 공통 기반 → **Phase B 블랙잭**(기준 구현 확립) → **C 바카라 → D 룰렛 → E 슬롯**.
3. 각 Phase 완료 게이트: 서버 테스트 전체 그린 + 클라 빌드 + 실기기(모바일) 실측 + 패리티 체크리스트.
4. 선행 문서의 P0 버그(BUG-01~05)는 Pixi 전환과 무관하게 최우선 유지.
